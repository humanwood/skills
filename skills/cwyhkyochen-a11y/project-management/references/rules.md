# 项目看板操作规则

## 0. 人员字段

### 创建人 (created_by)
**规则**:
- 记录项目的创建者（agent 名称或用户名称）
- 默认值：当前 agent 或操作用户
- 可手动指定：协作项目时指定其他创建人
- **创建后不可变更**（保持历史记录）

**示例**:
```markdown
| 项目名称 | 创建人 | 说明 |
|---------|--------|------|
| 客服话术对练系统 | yoyo | AI 助手创建 |
| PMP 备考 | kyo | 用户自己创建 |
| 日本旅行 | yoyo | AI 协助规划 |
```

### 最后更新人 (modified_by)
**规则**:
- 记录最后一次修改项目的人
- **自动更新**: 任何项目变更时自动更新为当前操作者
- 默认值：当前 agent 或操作用户
- 可手动指定：协作场景

**示例**:
```markdown
| 项目名称 | 创建人 | 最后更新人 | 说明 |
|---------|--------|-----------|------|
| 客服话术对练系统 | yoyo | kyo | kyo 更新了版本 |
| PMP 备考 | kyo | kyo | 用户自己维护 |
```

### 按人筛选
- "看看我创建了哪些项目" → 筛选 created_by=当前用户
- "看看 yoyo 负责的项目" → 筛选 created_by=yoyo
- "我最近改了哪些项目" → 筛选 modified_by=当前用户

## 1. 序号管理

**规则**:
- 新序号 = 当前最大序号 + 1
- 删除项目后不重新编号，保持历史序号
- 序号用于追踪项目历史，不是当前项目数量

**示例**:
```
当前最大序号：5
新增项目 → 序号：6

删除序号 3 的项目后：
剩余序号：1, 2, 4, 5（不重新编号）
新增项目 → 序号：6（继续递增）
```

## 2. 变更日志（独立文件）

**文件**: `projects/CHANGELOG.md`（独立于 PROJECTS.md）

**规则**:
- 所有操作必须记录变更日志
- 格式：`| 日期 | 操作类型 | 项目名称 | 更新人 | 说明 |`
- 按日期倒序排列（最新在前）

**操作类型**:
- 新增项目
- 状态变更
- 版本发布
- 移除项目
- 创建看板
- 新增类型

**示例**:
```markdown
# 📝 项目变更日志 (Project Changelog)

> 独立记录所有项目变更历史

---

## 变更历史

| 日期 | 操作 | 项目 | 更新人 | 说明 |
|------|------|------|--------|------|
| 2026-02-22 | 版本发布 | Gemini Agent Client | yoyo | v1.1.1 发布：自动标题 + 复制按钮 |
| 2026-02-21 | 状态变更 | Gemini Agent Client | kyo | 迭代中→暂停中，等待需求确认 |
| 2026-02-20 | 新增项目 | PMP 备考 | yoyo | 学习计划启动，目标 3 个月通过 |
```

**优势**:
- PROJECTS.md 保持简洁，只保留当前状态
- CHANGELOG.md 完整历史，可独立查看/归档
- 避免主文件过大

## 3. 日期格式

**规则**:
- 统一使用 `YYYY-MM-DD` 格式
- 使用当天日期（不写时间）
- 变更日志和项目最后改动时间都要更新

**示例**:
```
✅ 正确：2026-02-22
❌ 错误：2026/02/22
❌ 错误：2026-2-22
❌ 错误：2026-02-22 14:30
```

## 4. 路径规范

**规则**:
- 使用相对路径
- 代码项目：`./项目名/` 或 `../项目名/`
- 无实体路径的项目（学习、旅行等）：用 `-`

**示例**:
```markdown
| 项目名称 | 位置 |
|---------|------|
| 客服话术对练系统 | `./cs-training/` |
| PMP 备考 | `-` |
| 日本旅行 | `-` |
| 旧项目迁移 | `../old-projects/cs-training/` |
```

## 5. 版本号规范

**规则**:
- 遵循语义化版本：`v主版本。次版本.修订号`
- 初始版本：v0.1 或 v1.0
- 格式：`v1.0`, `v1.1`, `v2.0`

**版本号规则**:
- **主版本号 (X.0.0)**: 重大功能变更、架构调整
- **次版本号 (0.X.0)**: 新增功能、向下兼容
- **修订号 (0.0.X)**: Bug 修复、小优化

**示例**:
```
v0.1 → 初始版本
v1.0 → 正式发布
v1.1 → 新增功能
v1.1.1 → Bug 修复
v2.0 → 重大更新
```

## 6. 状态变更规则

**状态流转**:
```
🟢 迭代中 → 🟡 暂停中 → 🟢 迭代中（恢复）
🟢 迭代中 → 🔴 已结束（完成/废弃）
🟡 暂停中 → 🔴 已结束（放弃）
```

**变更记录**:
- 必须记录变更原因
- 格式：`原状态→新状态，原因`

**示例**:
```markdown
| 日期 | 操作 | 项目 | 说明 |
|------|------|------|------|
| 2026-02-22 | 状态变更 | 项目 A | 迭代中→暂停中，等待需求确认 |
| 2026-02-21 | 状态变更 | 项目 B | 暂停中→已结束，需求取消 |
```

## 7. 快速链接

**规则**:
- 仅针对有 README 的代码项目
- 格式：`- [项目名称](./项目目录/README.md)`
- 无 README 的项目不添加链接

**示例**:
```markdown
## 🔗 快速链接

- [客服话术对练系统](./cs-training/README.md)
- [Gemini Agent Client](./gemini-agent-client/README.md)
```

## 8. 备份建议

**重大变更前建议备份**:
- 批量删除项目
- 大规模状态调整
- 看板结构重构

**备份方法**:
```bash
cp projects/PROJECTS.md projects/PROJECTS.md.backup.2026-02-22
```

---

## 9. 维护频率

**建议**:
- 项目变更时立即更新（实时）
- 每周审查一次看板状态
- 每月清理已结束项目（归档）

**责任人**:
- 维护人：AI 助手
- 审核人：项目负责人
